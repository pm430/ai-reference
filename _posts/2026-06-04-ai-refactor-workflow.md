## 1. 들어가며

리팩토링은 **"동작은 그대로, 코드는 깔끔하게"** 만드는 작업입니다. 하지만 실무에서는 우선순위에서 밀리고, 정작 시작하면 어디서부터 손대야 할지 막막한 경우가 많습니다. AI는 이 막막함을 줄여 주지만, 동시에 **통제 없이 맡기면 오히려 회귀 버그**를 만들 수 있습니다.

이 글은 실제로 작동한 **5단계 워크플로우**를 그대로 공유합니다. 단순히 "잘 되는 예시"가 아니라, AI에게 어떤 지시를 줬고 어디서 멈췄는지까지 모두 공개합니다.

---

## 2. Why: 리팩토링에 AI를 쓰면 좋은 이유

리팩토링은 AI의 강점이 가장 잘 맞는 작업 중 하나입니다.

1. **패턴 인식**: 중복 코드, 매직 넘버, 과도한 중첩 같은 코드 스멜을 빠르게 잡아냅니다.
2. **멋진 결과를 빠르게 보여줌**: "이런 구조가 가능하구나" 하는 영감을 줘서, 이후 작업의 기준선이 됩니다.
3. **수정 범위 가이드**: "이 함수만 단위 테스트 짜줘" 같은 좁은 지시로 회귀 위험을 통제할 수 있습니다.

하지만 모든 작업을 맡기면 안 됩니다. **3번 항목이 핵심**이며, 이 글의 절반은 "어디서 멈춰야 하는가"에 할애합니다.

---

## 3. How: 5단계 워크플로우

### Step 1. 범위 정하기 — 가장 작은 단위로

리팩토링의 첫 번째 실수: **"이 파일 전체 정리해줘"** 같은 큰 지시. 결과가 한 번에 나오면 검토할 분량이 너무 많고, 회귀가 발생해도 원인을 찾기 어렵습니다.

**규칙**: 한 번에 **하나의 함수 또는 하나의 책임**만 맡깁니다.

```text
사용자: payments/calc.py 의 calculate_total() 함수만 리팩토링해줘.
       다른 함수는 건드리지 말고, 시그니처(인자, 반환값)는 절대 바꾸지 마.
       변경 후 동작이 동일함을 확인할 수 있는 단위 테스트도 함께 작성해줘.
```

이 한 줄의 "시그니처 절대 바꾸지 마" + "단위 테스트도 함께" 가 안전판입니다.

---

### Step 2. Before — 리팩토링 대상 코드

다음은 실제 사례로 단순화한 예시입니다. 가격, 할인, 등급 정책이 한 함수에 뭉쳐 있어 읽기 어렵습니다.

```python
# payments/calc.py
def calculate_total(items, user):
    t = 0
    for i in items:
        if i['t'] == 'A':
            t += i['p'] * 0.9
        elif i['t'] == 'B':
            t += i['p'] * 0.8
        else:
            t += i['p']
    if user == 'gold':
        t *= 0.95
    elif user == 'silver':
        t *= 0.98
    return t
```

**문제점**:

- 매직 넘버 (`0.9`, `0.8`, `0.95`, `0.98`) 가 의미를 설명하지 않음
- 약어 (`t`, `i`, `p`) 가 의도를 가림
- 정책(카테고리 할인, 등급 할인)이 한 함수에 섞여 있어 테스트가 어려움

---

### Step 3. AI에게 보낸 프롬프트

```text
시스템: 너는 Python 리팩토링 전문가다. 다음 원칙을 따른다.
- 매직 넘버를 상수로 추출하고 의미 있는 이름을 붙인다.
- 약어를 풀 네임으로 바꾼다.
- 단일 책임 원칙(SRP)을 적용해 함수를 분리한다.
- 원본 동작을 그대로 유지한다 (시그니처 동일).
- 변경된 동작을 검증할 단위 테스트를 함께 작성한다.

사용자: 아래 calculate_total() 함수를 리팩토링해줘.
       표준 라이브러리만 사용하고 외부 의존성은 추가하지 마.

[paste code]
```

**프롬프트 설계 포인트**:

| 요소 | 이유 |
| --- | --- |
| 시스템에 원칙 명시 | 매번 같은 품질의 결과 |
| "시그니처 동일" 명시 | 호출부 회귀 방지 |
| "외부 의존성 추가 금지" | 의존성 폭증 방지 |
| "단위 테스트 함께" | 검증 책임도 함께 위임 |

---

### Step 4. After — AI가 만든 결과

```python
# payments/calc.py
from typing import TypedDict

CATEGORY_DISCOUNT = {
    'A': 0.10,  # 10% 할인
    'B': 0.20,  # 20% 할인
    'default': 0.0,
}

GRADE_DISCOUNT = {
    'gold': 0.05,
    'silver': 0.02,
    'default': 0.0,
}


class Item(TypedDict):
    type: str
    price: float


def _apply_category_discount(price: float, category: str) -> float:
    return price * (1 - CATEGORY_DISCOUNT.get(category, CATEGORY_DISCOUNT['default']))


def _apply_grade_discount(total: float, grade: str) -> float:
    return total * (1 - GRADE_DISCOUNT.get(grade, GRADE_DISCOUNT['default']))


def calculate_total(items: list[Item], user: str) -> float:
    subtotal = sum(_apply_category_discount(item['price'], item['type']) for item in items)
    return _apply_grade_discount(subtotal, user)
```

같은 동작을 하면서도:

- **상수 추출**: 할인율이 코드만 봐도 의미가 명확해짐
- **함수 분리**: 카테고리 할인과 등급 할인이 독립적으로 테스트 가능
- **타입 힌트**: `TypedDict` 로 입력 형태가 자명해짐
- **가독성**: 본문이 3줄로 줄어 의도가 한 눈에 들어옴

---

### Step 5. 검증 3단계 체크리스트

AI가 만든 코드를 **그대로 머지하지 마세요**. 다음 3단계를 반드시 거치세요.

#### 1) 단위 테스트 실행

```bash
pytest payments/test_calc.py -v
```

기존 테스트가 있다면 모두 통과해야 하고, 없다면 AI에게 함께 작성하게 한 테스트로 **동작 동등성**을 확인합니다.

#### 2) 회귀 테스트 — 호출부 동작 확인

```bash
# 기존 입출력 사례를 모아 둔 골든 테스트가 있다면 가장 확실
pytest tests/golden/test_payments_golden.py
```

이 단계에서 **사양과 다른 결과가 한 건이라도 나오면 즉시 롤백**합니다. "그냥 고치면 되지"가 가장 위험한 판단입니다.

#### 3) 코드 리뷰 — 사람이 한 번 더 본다

특히 확인할 지점:

- 시그니처(인자, 반환 타입) 변경 여부
- 외부 의존성 추가 여부
- 매직 넘버/주석 처리되지 않은 하드코딩 잔존
- 예외 동작 (빈 리스트, 알 수 없는 카테고리/등급)

---

## 4. 어디서 멈춰야 하는가 — 경계

AI가 잘하는 영역과 **사람이 반드시 결정해야 하는 영역**을 구분합니다.

| 영역 | AI에게 맡길까? | 이유 |
| --- | --- | --- |
| 변수명·함수명 개선 | ✅ | 가역 비용, 자동 리뷰 가능 |
| 중복 코드 추출 | ✅ | 기계적 패턴 인식 |
| 매직 넘버 → 상수 | ✅ | 의미만 잘 설명하면 됨 |
| 라이브러리 교체/도입 | ❌ | 의존성·라이선스 검토 필요 |
| 아키텍처 변경 (함수 → 클래스, 모듈 분리) | ⚠️ 신중 | 작은 PR 단위로 점진적으로 |
| 성능 최적화 | ❌ | 프로파일링 후 결정 |
| 정책/비즈니스 규칙 변경 | ❌ 절대 | 제품 결정은 사람의 영역 |

**가장 중요한 한 줄**: AI에게 맡기는 작업의 **가역성(reversibility)** 을 항상 따져보세요. 되돌리기 쉬운 변경은 적극적으로, 되돌리기 어려운 변경은 신중하게.

---

## 5. 흔한 안티패턴

- **"전체 코드베이스 리팩토링해줘"** — 한 번에 머지할 수 없는 분량이 나옵니다. PR이 3000줄이 되면 리뷰가 불가능해집니다.
- **"테스트 없이 리팩토링만"** — 동작 동등성을 입증할 수단이 없으면, 추후 회귀가 발생했을 때 "이게 내 변경 때문인가?" 를 알 수 없습니다.
- **AI가 추천한 라이브러리를 그대로 추가** — 의존성은 작은 결정 같아 보여도 운영 비용·보안 이슈로 확장됩니다.
- **리팩토링과 기능 변경을 한 PR에 섞기** — 리뷰어와 본인 모두를 위한 격리 원칙입니다. 되돌릴 때도 통째로 롤백해야 합니다.

---

## 6. 한 단계 더 — 자동화 도구 연결

리팩토링을 한 번에 끝내려 하지 말고, **계속 작은 단위로 도와주는 환경**을 만드세요.

- **Pre-commit hook**: `ruff` / `black` 같은 포매터를 걸어 두면, AI 결과의 스타일 차이도 자동 흡수됩니다.
- **CI에 단위 테스트 필수**: 머지 전 무조건 실행되는 환경이 있어야 AI 결과를 안심하고 받아들일 수 있습니다.
- **Claude Code Hooks**: 특정 명령(예: `npm test`)을 사전/사후에 강제 실행하도록 설정해 두면, AI가 "테스트도 돌려본 결과" 라는 맥락을 자동으로 얻게 됩니다.

이 도구들은 *하네스 엔지니어링* 의 사상과 직결됩니다. AI가 안전하게 움직일 수 있는 울타리를 코드 차원에서 만들어 두는 것이죠.

---

## 7. 마치며

AI는 리팩토링을 **더 빠르고 더 즐겁게** 만들어 줍니다. 하지만 **결정권과 책임은 여전히 사람에게** 있습니다. 핵심은 다음 세 가지로 요약됩니다.

1. **작게** — 한 번에 하나씩, 시그니처는 유지
2. **검증** — 단위 테스트 + 회귀 테스트 + 사람 리뷰의 3단계
3. **되돌리기 쉬운 변경만** 위임 — 가역성을 기준으로 경계 설정

이 원칙만 지켜도, AI는 막강한 리팩토링 조수가 됩니다.
