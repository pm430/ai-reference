MCP 글을 쓰다 보면 설치, 연결, 사용 예시는 많은데 보안 이야기는 늘 마지막에 붙습니다. 그런데 2026년 7월 17일 기준으로는 이 순서가 뒤집혀야 합니다. MCP 서버가 이메일, 문서, 데이터베이스, 이슈 트래커 같은 실제 자산에 연결되기 시작하면서 **인증과 권한 설계가 사실상 기본값**이 됐기 때문입니다.

> MCP는 "도구를 더 쉽게 붙이는 표준"이지만, 동시에 "권한을 더 신중하게 설계해야 하는 표준"이기도 합니다.

---

## 1. 왜 지금 MCP 보안이 핵심 주제가 됐나

초기 MCP 사용은 대체로 로컬 파일이나 단순 도구 연결에 머물렀습니다. 하지만 지금은 다음 같은 연결이 흔합니다.

- 이메일
- 문서 저장소
- 이슈 트래커
- 데이터베이스
- 내부 운영 시스템

이 단계부터는 도구 연결이 아니라 **권한 위임** 문제가 됩니다.

Model Context Protocol 공식 보안 문서도 이 점을 분명히 합니다. 사용자 데이터, 관리 작업, 동의가 필요한 API, 기업 환경 접근 제어, 사용자별 추적이 필요한 경우에는 authorization이 강하게 권장됩니다.

---

## 2. 언제 authorization이 사실상 필수인가

MCP 문서는 authorization이 "optional"이라고 말하지만, 실무에서는 아래 중 하나만 해당해도 거의 필수라고 봐야 합니다.

### 2.1 사용자별 데이터에 접근할 때

- Gmail
- Drive
- Notion
- 내부 문서

이런 경우 "누가 어떤 데이터에 접근했는가"를 추적할 수 있어야 합니다.

### 2.2 파괴적 작업이 가능할 때

- 이슈 생성/수정/삭제
- DB 쓰기
- 배포
- 설정 변경

읽기 전용과 쓰기 권한을 분리하지 않으면 사고가 납니다.

### 2.3 엔터프라이즈 환경일 때

감사 로그, 권한 최소화, 사용자 동의, 토큰 회수 같은 요구사항이 생깁니다.

---

## 3. MCP authorization의 큰 구조

MCP specification과 공식 튜토리얼은 OAuth 2.1 기반 모델을 설명합니다.

핵심 역할은 단순합니다.

| 역할 | 의미 |
| --- | --- |
| MCP client | 사용자를 대신해 서버에 접근하는 쪽 |
| MCP server | 보호된 리소스를 노출하는 쪽 |
| Authorization server | 로그인, 동의, 토큰 발급을 담당 |

즉, MCP 서버는 단순 함수 호출 엔드포인트가 아니라 **보호된 자원 서버**처럼 다뤄야 합니다.

이 구조가 중요한 이유는 "에이전트가 알아서 붙는다"는 편의성과 "누가 어떤 권한으로 붙는가"를 분리해 주기 때문입니다.

---

## 4. 최소한 지켜야 할 보안 체크리스트

MCP 서버를 직접 만들거나 운영한다면 아래는 거의 기본 체크리스트입니다.

### 4.1 최소 권한

- 읽기만 필요한데 쓰기 권한을 주지 말 것
- 서버 단위가 아니라 도구 단위로도 권한 범위를 생각할 것

### 4.2 사용자/행위 감사 가능성

- 누가 호출했는가
- 어떤 도구를 실행했는가
- 어떤 자원에 접근했는가

감사 흔적이 없으면 사고 후 원인 규명이 어렵습니다.

### 4.3 승인과 자동화의 분리

아래 작업은 자동 승인보다 사람 승인 단계를 두는 편이 낫습니다.

- 외부 발송
- 삭제
- 금전/계약 관련 변경
- 운영 설정 변경

### 4.4 프롬프트 인젝션과 간접 입력 오염 대비

MCP 보안 문서는 전통적인 웹 보안만이 아니라, 모델이 읽는 외부 컨텐츠 자체가 공격 표면이 될 수 있음을 전제로 합니다. 즉 도구 응답, 문서, 웹페이지 내용이 에이전트 행동을 바꾸지 못하게 해야 합니다.

### 4.5 토큰과 세션 수명 관리

- 장기 토큰 남용 방지
- 회수 가능성
- 세션 만료
- 사용자 단위 추적

---

## 5. 로컬 MCP와 원격 MCP를 같은 감각으로 보면 안 된다

개발자들은 로컬 MCP를 많이 다루다 보니 감각이 무뎌지기 쉽습니다. 하지만 둘은 위험도가 다릅니다.

| 환경 | 주된 위험 |
| --- | --- |
| 로컬 전용 MCP | 파일 오염, 과도한 권한, 실수 |
| 원격/사내 MCP | 계정 오남용, 데이터 유출, 감사 실패 |
| 외부 SaaS MCP | 인증 누락, 동의 범위 과대, 토큰 관리 |

그래서 원격 MCP는 "잘 붙는다"보다 "잘 끊기고 잘 제한되며 잘 추적되는가"가 더 중요합니다.

---

## 6. 실무에서 자주 하는 나쁜 설계

### 6.1 서버 하나에 모든 권한 몰아주기

편해서 그렇게 만들기 쉽지만, 나중에 감사와 권한 분리가 거의 불가능해집니다.

### 6.2 읽기/쓰기 도구를 구분하지 않기

같은 서버에 있어도 도구 레벨에서 위험도가 완전히 다를 수 있습니다.

### 6.3 사용자 동의 범위를 숨기기

특히 팀 내 배포에서 "일단 편하니까 전체 권한"으로 가면 나중에 회수 비용이 커집니다.

### 6.4 자동화 성공률만 보고 보안 로그를 안 남기기

성공률이 높아도 사고 나면 대응이 안 됩니다.

---

## 7. 지금 블로그에서 빠져 있던 이유와, 이제 채워야 하는 이유

이 블로그는 지금까지 MCP를 "어떻게 붙일까" 쪽에서 잘 다뤘습니다.

- MCP가 왜 중요한가
- Smithery로 어떻게 찾고 설치하는가
- Python으로 서버를 직접 어떻게 만드는가

하지만 현시점의 큰 공백은 **보안과 권한 모델**이었습니다. 그리고 이건 사소한 보완이 아니라, MCP가 장난감 단계를 넘어 실서비스/실조직 워크플로우로 들어가면서 가장 먼저 요구되는 주제입니다.

---

## 8. 마치며

MCP는 에이전트의 손과 발을 늘려 주는 표준입니다. 그래서 보안 설계는 선택이 아니라 기본값이어야 합니다.

정리하면:

1. 사용자 데이터나 쓰기 작업이 있으면 authorization을 기본으로 본다
2. 최소 권한, 감사 가능성, 승인 단계, 토큰 관리부터 설계한다
3. 로컬 도구 연결 감각을 원격 시스템 연결에 그대로 가져가지 않는다

도구를 더 많이 붙일수록, "무엇을 할 수 있나"보다 "무엇을 못 하게 막았나"가 더 중요해집니다.
