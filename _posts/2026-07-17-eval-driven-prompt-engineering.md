프롬프트 엔지니어링이 여전히 중요하냐는 질문을 자주 받습니다. 답은 "그렇다, 하지만 방식이 달라졌다"입니다. 2026년 7월 17일 기준 공식 문서들의 공통 메시지는 분명합니다. **프롬프트는 반드시 evals 위에서 개선해야 한다**는 점입니다.

> 좋은 프롬프트는 한 번에 떠오르는 문장이 아니라, 평가를 통과하며 살아남은 버전입니다.

---

## 1. 왜 "감으로 고치는 프롬프트"가 한계에 닿았나

예전에는 프롬프트를 조금 바꾸고 예시 몇 개를 돌려 본 다음 "이제 좀 좋아졌네" 하고 넘어가도 됐습니다. 하지만 에이전트형 워크플로우에서는 이런 방식이 오래 버티지 못합니다.

- 입력 변형이 훨씬 많아졌고
- 출력 형식 요구가 더 엄격해졌으며
- 도구 사용 여부까지 품질에 포함되고
- 모델 교체 주기가 빨라졌기 때문입니다

Anthropic도 프롬프트 엔지니어링 가이드에서 먼저 성공 기준, 실험 수단, 초안 프롬프트가 있어야 한다고 전제합니다. OpenAI 역시 모델이 비결정적이기 때문에 전통적인 테스트만으로는 부족하고, evals가 필요하다고 설명합니다.

---

## 2. 평가 기반 프롬프트 엔지니어링이란 무엇인가

핵심은 단순합니다.

```text
프롬프트 초안
-> 테스트셋 실행
-> 점수/실패 유형 확인
-> 프롬프트 수정
-> 같은 테스트셋 재실행
-> 이전 버전과 비교
```

이 흐름이 없으면 프롬프트 수정은 사실상 취향 문제에 머뭅니다.

### 감 기반 방식

- "이번 답변은 더 좋아 보인다"
- "좀 더 친절해진 것 같다"
- "왠지 구조가 나아졌다"

### 평가 기반 방식

- 20개 케이스 중 JSON 포맷 성공률이 80%에서 95%로 상승
- 필수 항목 누락이 6건에서 1건으로 감소
- 툴 호출 누락이 4건에서 0건으로 감소

후자가 실무에서 살아남습니다.

---

## 3. 프롬프트를 고치기 전에 먼저 해야 하는 것

Anthropic의 최신 문서는 이 지점을 아주 명확하게 짚습니다.

1. 성공 기준 정의
2. 그 기준을 실험으로 확인할 방법 마련
3. 개선할 첫 프롬프트 초안 확보

이 셋이 없으면 프롬프트를 고쳐도 비교 자체가 안 됩니다.

예를 들어 고객 지원 분류 프롬프트를 만든다면:

| 항목 | 예시 |
| --- | --- |
| 목표 | 문의를 `billing`, `technical`, `refund` 중 하나로 분류 |
| 실패 기준 | 잘못된 라벨, 근거 누락, JSON 포맷 오류 |
| 데이터셋 | 실제 문의 30건 + 경계 사례 10건 |
| 점수화 | 정답 라벨 일치율, 필드 완성률 |

이렇게 해야 "어떻게 프롬프트를 바꿀지"가 보입니다.

---

## 4. 무엇을 프롬프트로 해결하고, 무엇은 다른 레이어로 올려야 하나

여기서 중요한 함정이 하나 있습니다. 모든 문제를 프롬프트 수정으로 해결하려고 들면 안 됩니다.

Anthropic 문서도 말하듯, 어떤 실패는 모델 선택이나 비용/지연 구조 문제일 수 있습니다.

### 프롬프트로 해결하기 좋은 문제

- 출력 형식이 흔들린다
- 응답 톤이 불안정하다
- 필수 단계가 누락된다
- 도구 사용을 주저한다

### 프롬프트보다 다른 해법이 나은 문제

- 너무 느리다
- 너무 비싸다
- 모델 자체 추론력이 부족하다
- 컨텍스트가 과하게 길다
- 보안/권한 문제가 있다

즉 프롬프트는 강력하지만, **항상 1순위 해법은 아닙니다**.

---

## 5. 가장 효과적인 프롬프트 수정 패턴

Anthropic의 prompting best practices를 실무 흐름으로 재구성하면 아래 다섯 가지가 가장 자주 먹힙니다.

### 5.1 지시를 더 명확하게 쓴다

- 결과 형식 명시
- 금지 조건 명시
- 단계 순서 명시

모호한 문장을 줄이는 것만으로도 흔들림이 크게 줄어듭니다.

### 5.2 예시를 넣는다

Few-shot 예시는 형식과 톤을 안정시키는 가장 빠른 방법입니다.

특히 아래에 유리합니다.

- JSON 출력
- 분류 작업
- 요약 템플릿
- 리뷰 포맷

### 5.3 XML/섹션 구조를 쓴다

Anthropic은 여전히 구조적 구분을 강하게 권장합니다.

```xml
<instructions>...</instructions>
<context>...</context>
<question>...</question>
```

이 방식은 지시, 배경, 입력을 섞지 않게 해줍니다.

### 5.4 툴 사용을 더 노골적으로 지시한다

최신 Claude 계열은 명시적 도구 지시에서 더 잘 동작합니다. "필요하면 도구를 써라"보다 "먼저 파일 3개를 읽고, 그다음 수정하고, 마지막에 테스트를 실행하라"가 낫습니다.

### 5.5 병렬성과 장기 상태를 고려한다

최근 문서들은 병렬 도구 호출, 장기 상태 추적, 서브에이전트 오케스트레이션까지 프롬프트/스캐폴딩 문제로 다룹니다. 즉 프롬프트 엔지니어링은 더 이상 문장 꾸미기가 아니라 **작업 구조를 언어로 설계하는 일**에 가깝습니다.

---

## 6. 가장 실용적인 프롬프트 개선 루프

현실적인 루프는 아래 정도면 충분합니다.

1. 현재 프롬프트를 변수화한다
2. 테스트 케이스를 20~50개 준비한다
3. 실패 유형을 3개 정도로 분류한다
4. 한 번에 한 가지만 수정한다
5. 이전 버전과 나란히 비교한다

중요한 점은 **한 번에 여러 가설을 섞지 않는 것**입니다.

나쁜 예:

- 예시 추가
- 출력 형식 수정
- 모델 교체
- 컨텍스트 순서 변경

이걸 한 번에 하면 뭐가 효과 있었는지 모릅니다.

좋은 예:

- 이번에는 예시만 추가
- 다음에는 XML 구조만 적용
- 다음에는 툴 호출 지시만 강화

---

## 7. 콘솔 도구를 활용하는 방식도 생각보다 강력하다

Anthropic의 Evaluation Tool은 단순 데모가 아니라 프롬프트 버전 비교와 테스트 케이스 반복에 꽤 실용적입니다.

- 변수 기반 프롬프트 작성
- 테스트 케이스 수동/자동 생성
- 프롬프트 버전 비교
- 품질 등급 비교

즉, "프롬프트를 복붙해가며 수동으로 실험"하던 방식에서 벗어나는 출발점으로 좋습니다.

---

## 8. 마치며

프롬프트 엔지니어링은 끝난 게 아닙니다. 다만 이제는 **평가 없는 프롬프트 엔지니어링이 설 자리가 줄어든 것**에 가깝습니다.

정리하면:

1. 먼저 성공 기준을 정한다
2. 테스트셋을 만든다
3. 프롬프트는 한 번에 하나씩만 바꾼다
4. 이전 버전과 비교한다
5. 프롬프트로 안 풀리는 문제는 모델/컨텍스트/도구/권한 레이어로 올린다

이 습관만 잡혀도 프롬프트 품질은 훨씬 덜 흔들립니다.
