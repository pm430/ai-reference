2026년 7월 17일 기준으로 AI 에이전트를 제대로 운영하려면, 프롬프트보다 먼저 **평가 체계**를 세워야 합니다. 최근 공식 문서들도 같은 방향을 가리킵니다. OpenAI는 에이전트 평가를 위한 traces, graders, datasets, eval runs를 전면에 두고 있고, Anthropic은 프롬프트를 고치기 전에 먼저 성공 기준과 실험 방법을 정하라고 말합니다.

> 에이전트 시대의 첫 번째 질문은 "어떻게 프롬프트를 쓸까?"가 아니라 "무엇을 성공으로 볼까?"입니다.

---

## 1. 왜 갑자기 evals가 중요해졌나

기존 챗봇은 한 번의 응답 품질만 봐도 되는 경우가 많았습니다. 하지만 에이전트는 다릅니다.

- 여러 단계로 작업을 분해합니다.
- 도구를 호출합니다.
- 상태를 기억하거나 저장합니다.
- 중간에 실패하면 다시 시도합니다.
- 최종 산출물까지 완성해야 합니다.

이 구조에서는 단순히 "답이 그럴듯한가"만으로 품질을 판단할 수 없습니다. 예를 들어:

| 질문 | 챗봇 시절 | 에이전트 시절 |
| --- | --- | --- |
| 답변이 맞는가 | 중요 | 여전히 중요 |
| 올바른 도구를 썼는가 | 덜 중요 | 매우 중요 |
| 불필요한 단계가 많지 않은가 | 덜 중요 | 중요 |
| 재시도 후 더 좋아졌는가 | 거의 안 봄 | 중요 |
| 같은 요청에 일관되게 동작하는가 | 중요 | 더 중요 |

OpenAI도 최근 가이드에서 에이전트 품질 개선을 위해 traces, graders, datasets, eval runs를 함께 쓰라고 설명합니다. Anthropic 역시 프롬프트 개선 전에 성공 기준과 경험적 테스트 수단이 있어야 한다고 못 박습니다.

---

## 2. evals를 너무 어렵게 생각할 필요는 없다

많은 팀이 evals를 거창한 벤치마크 프로젝트로 오해합니다. 실제로는 아래 네 가지만 있으면 출발할 수 있습니다.

1. **성공 기준**
2. **대표 테스트셋**
3. **채점 방식**
4. **비교 가능한 버전 관리**

이 네 가지를 갖추면, 프롬프트를 바꾸든 모델을 바꾸든 도구 연결을 바꾸든 개선 여부를 확인할 수 있습니다.

### 2.1 성공 기준

Anthropic 문서가 강조하는 핵심은 `Specific` 과 `Measurable` 입니다.

- 나쁜 기준: "답변 품질이 좋다"
- 좋은 기준: "출처 2개 이상을 포함하고, 핵심 질문 3개 중 2개 이상에 답한다"

에이전트라면 여기에 실행 기준도 들어가야 합니다.

- 필요한 도구를 실제로 호출했는가
- 금지된 작업은 하지 않았는가
- 지정한 출력 포맷을 지켰는가
- 시간/토큰/비용 한도를 넘지 않았는가

### 2.2 테스트셋

테스트셋은 많을 필요보다 **대표성**이 중요합니다.

- 자주 들어오는 정상 요청
- 애매한 요청
- 실패하기 쉬운 요청
- 금지/거절이 필요한 요청

처음에는 20개만 있어도 충분합니다. 중요한 것은 "현실에서 자주 깨지는 패턴"이 들어 있느냐입니다.

### 2.3 채점 방식

채점은 보통 세 층으로 나뉩니다.

| 방식 | 용도 |
| --- | --- |
| 규칙 기반 | JSON 형식, 필수 필드, 금지어, 툴 호출 여부 |
| 사람 검토 | 초기 기준 만들기, 애매한 케이스 |
| LLM grader | 대규모 비교, 서술형 응답 품질 추세 확인 |

초기에는 규칙 기반 + 사람 검토 조합이 가장 실용적입니다.

### 2.4 버전 비교

같은 테스트셋으로 아래를 비교할 수 있어야 합니다.

- 프롬프트 A vs 프롬프트 B
- 모델 A vs 모델 B
- 도구 연결 전 vs 후
- 안전장치 추가 전 vs 후

이게 안 되면 개선이 아니라 느낌 싸움이 됩니다.

---

## 3. 에이전트 evals는 일반 소프트웨어 테스트와 어떻게 다른가

일반 테스트는 대개 결정적입니다. 같은 입력이면 같은 결과가 기대됩니다. 하지만 LLM 시스템은 본질적으로 변동성이 있습니다.

그래서 evals는 "정답이 1개"인 시험이라기보다 **허용 가능한 행동 범위를 측정하는 실험**에 가깝습니다.

```text
요청
-> 에이전트 실행
-> 중간 trace 저장
-> 결과물 + 도구 호출 기록 수집
-> 규칙/그레이더/사람 검토로 점수화
-> 이전 버전과 비교
```

여기서 trace가 특히 중요합니다. 결과만 보면 왜 실패했는지 모르는 경우가 많기 때문입니다.

---

## 4. 가장 실용적인 시작 순서

처음부터 플랫폼 기능을 다 쓰지 않아도 됩니다. 아래 순서가 현실적입니다.

### 1단계: 실패 사례 수집

- 실제 사용 로그에서 자주 망가지는 케이스 10~20개 추출
- "좋은 결과"와 "나쁜 결과" 예시를 나란히 저장

### 2단계: 성공 기준 문장화

- 각 케이스에 대해 무엇이 통과인지 한 줄로 적기
- 가능하면 정량화하기

### 3단계: 최소 채점기 만들기

- 포맷 검사
- 필수 항목 검사
- 금지 항목 검사

### 4단계: 변경 전후 비교하기

- 프롬프트 수정
- 모델 교체
- 도구 추가
- 컨텍스트 변경

무엇을 바꾸든 같은 테스트셋으로 다시 돌립니다.

### 5단계: 회귀 검증 습관 만들기

한 번 좋아졌다고 끝이 아닙니다. 새 모델이나 새 도구를 붙일 때마다 회귀가 납니다.

---

## 5. 무엇을 eval로 해결하고, 무엇은 다른 방법으로 해결해야 하나

Anthropic의 문서가 좋은 균형점을 줍니다. 모든 문제를 프롬프트 엔지니어링으로 풀지 말라는 점입니다.

실제로는 아래처럼 나뉩니다.

| 문제 | 우선 해법 |
| --- | --- |
| 출력 형식이 흔들림 | 프롬프트/예시 |
| 도구를 안 쓰거나 엉뚱하게 씀 | 툴 스캐폴딩, 명시적 지시, 권한 설계 |
| 너무 느리거나 비쌈 | 모델 교체, 컨텍스트 축소 |
| 중간에 방향을 잃음 | 상태 저장, 체크포인트, 루프 설계 |
| 위험한 행동을 함 | 하네스, 승인, 권한, 차단 정책 |

evals의 역할은 "모든 걸 직접 해결"하는 것이 아니라, **어느 레이어를 고쳐야 하는지 보이게 하는 것**입니다.

---

## 6. 마치며

에이전트를 잘 쓰는 팀은 프롬프트를 더 멋지게 쓰는 팀이 아니라, **개선을 측정할 수 있는 팀**입니다. 그리고 그 출발점이 evals입니다.

처음부터 거대한 평가 플랫폼을 만들 필요는 없습니다. 오늘 당장 아래만 시작해도 충분합니다.

1. 자주 실패하는 요청 20개 모으기
2. 성공 기준 한 줄씩 적기
3. 포맷/필수 항목 검사 만들기
4. 프롬프트나 모델을 바꿀 때마다 같은 셋으로 다시 비교하기

이 네 단계만 갖춰도 "감으로 개선하는 상태"에서 벗어날 수 있습니다.
