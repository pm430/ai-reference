## 1. Loop Engineering이 왜 갑자기 나왔나

최근 Boris Cherny, Peter Steinberger 같은 AI 코딩 도구 사용자들이 "이제 코딩 에이전트에 직접 프롬프트하지 말고, 에이전트를 프롬프트하는 루프를 설계하라"는 식의 이야기를 많이 하고 있습니다.

처음 들으면 부담스럽습니다. Prompt Engineering, Context Engineering, Harness Engineering도 따라가야 하는데 이제 Loop Engineering까지 알아야 하나 싶죠. 하지만 영상의 핵심은 오히려 단순합니다.

> Loop Engineering은 **초기 지시 이후 반복되는 개선·검증·재시도를 사람이 매번 입력하지 않도록 만드는 설계**입니다.

즉, 프롬프트가 사라지는 것이 아닙니다. 사람이 매 반복마다 "다시 고쳐줘", "검증해줘", "부족한 부분 보완해줘"라고 입력하던 일을 시스템이 대신하게 만드는 것입니다.

## 2. Ralph Wiggum Loop, Harness Engineering, Loop Engineering의 차이

비슷해 보이는 개념을 구분하면 이해가 쉬워집니다.

| 개념 | 초점 | 한 줄 요약 |
|---|---|---|
| **Ralph Wiggum Loop** | 진행 | 작업을 잘게 쪼개고, 다음 작업으로 계속 넘어가게 함 |
| **Harness Engineering** | 제약 | 에이전트가 해도 되는 일과 하면 안 되는 일을 정의함 |
| **Loop Engineering** | 완성도 | 결과물을 관찰·평가·수정하면서 목표 품질까지 반복 개선함 |

Ralph Wiggum Loop는 긴 작업을 멈추지 않고 이어가기 위한 기법에 가깝습니다. 예전에는 컨텍스트 한계가 더 컸고, 에이전트가 한 번에 오래 달리기 어려웠기 때문에 작업을 작게 나누고 결과를 디스크에 저장한 뒤 새 컨텍스트에서 다음 작업을 수행하는 방식이 유용했습니다.

Harness Engineering은 루프보다 넓은 개념입니다. "이 파일은 수정하지 말 것", "테스트를 반드시 통과시킬 것", "외부 API 호출은 승인 전 실행하지 말 것"처럼 작업의 경계와 규칙을 세웁니다.

Loop Engineering은 그 위에서 **결과의 품질을 계속 올리는 반복 구조**에 집중합니다. 행동하고, 관찰하고, 판단하고, 다시 행동하는 구조입니다.

```text
목표 입력
  ↓
작업 수행
  ↓
결과 관찰
  ↓
평가 / 비평 / 점수화
  ↓
수정 방향 결정
  ↓
다시 작업 수행
  ↓
종료 조건 충족 시 사람에게 보고
```

## 3. "프롬프트하지 말라"는 말의 실제 의미

Loop Engineering을 오해하면 "처음부터 아무 프롬프트도 쓰지 않아야 한다"로 받아들이기 쉽습니다. 하지만 실제 의미는 다릅니다.

초기 목표 프롬프트는 여전히 필요합니다.

```text
이 기능을 구현해줘.
단, 기존 결제 플로우를 깨지 말고,
테스트와 타입 체크를 통과해야 하며,
변경 후 스스로 리뷰해서 취약점을 보완해줘.
```

Loop Engineering이 줄이려는 것은 그 다음부터 사람이 계속 넣는 n번째 프롬프트입니다.

```text
1차 결과: 여기 테스트 빠졌어. 추가해줘.
2차 결과: 기존 케이스가 깨졌어. 다시 확인해줘.
3차 결과: 타입 에러가 남았어. 고쳐줘.
4차 결과: 리뷰 관점에서 리스크를 찾아줘.
```

이 반복을 사람이 매번 입력하지 않도록, 루프 안에 **검증 기준·비평 방식·종료 조건**을 넣는 것이 핵심입니다.

---

## 4. 언제 쓰면 좋은가

Loop Engineering은 모든 작업에 필요한 기법이 아닙니다. 오히려 작은 작업에 억지로 적용하면 토큰과 시간만 낭비할 수 있습니다.

잘 맞는 경우는 다음과 같습니다.

- 큰 코드베이스에 새 기능을 넣어야 하고, 기존 기능과의 충돌을 계속 확인해야 할 때
- UI, 문서, 디자인 산출물처럼 한 번에 만족스러운 결과가 잘 나오지 않고 반복 개선이 필요한 때
- 테스트 작성, 리뷰, 리팩토링, 회귀 검증을 한 묶음으로 자동화하고 싶을 때
- 명확한 평가 기준이나 종료 조건을 줄 수 있을 때

반대로 다음 작업에는 과합니다.

- 단순 파일 수정
- 한 번의 프롬프트로 충분히 끝나는 작은 기능
- 평가 기준이 너무 모호해서 AI가 스스로 품질을 판단하기 어려운 작업
- 토큰 비용과 실행 시간이 더 중요한 개인 실험

핵심은 "유행하는 기법인가"가 아니라 **작업 규모에 맞는가**입니다.

## 5. 좋은 루프에 들어가야 할 요소

루프를 만들 때는 다음 다섯 가지가 있어야 합니다.

### 5.1 목표

무엇을 완성해야 하는지 명확해야 합니다.

```text
목표: 기존 상품 상세 페이지에 A/B 테스트 배너 기능을 추가한다.
```

### 5.2 관찰 방법

작업 결과를 어떻게 볼 것인지 정해야 합니다.

```text
- 테스트 실행 결과
- 빌드 결과
- 스크린샷 비교
- 로그 분석
- 린트 / 타입 체크
```

### 5.3 평가 기준

무엇이 좋아졌는지 판단할 기준이 필요합니다.

```text
- 모든 기존 테스트 통과
- 신규 테스트 추가
- 접근성 오류 없음
- 기존 API 계약 유지
- UI 스냅샷 차이가 의도한 영역에만 존재
```

### 5.4 수정 규칙

평가 결과를 다음 작업에 어떻게 반영할지 정합니다.

```text
실패한 테스트를 먼저 고친다.
그 다음 타입 에러를 고친다.
마지막으로 중복 코드와 이름을 정리한다.
```

### 5.5 종료 조건

루프가 언제 멈춰야 하는지 명시해야 합니다.

```text
- 테스트 / 빌드 / 린트가 모두 통과하면 종료
- 최대 5회 반복 후 사람에게 보고
- 같은 오류가 2회 이상 반복되면 중단하고 원인 분석 보고
```

종료 조건이 없으면 에이전트는 불필요하게 오래 돌거나, 반대로 적당한 결과에서 멈추지 못할 수 있습니다.

## 6. 실전 예시: 기능 구현 루프

대규모 코드베이스에서 기능 하나를 넣는다면 루프는 이렇게 설계할 수 있습니다.

```text
목표:
주문 상세 화면에 환불 상태 배지를 추가한다.

루프:
1. 관련 파일과 테스트를 찾는다.
2. 최소 변경으로 기능을 구현한다.
3. 타입 체크와 단위 테스트를 실행한다.
4. 실패하면 원인을 분류하고 수정한다.
5. 기존 주문 상태와 충돌하는 케이스를 스스로 리뷰한다.
6. 필요한 테스트가 없으면 추가한다.
7. 모든 검증이 통과하거나, 같은 실패가 2회 반복되면 보고한다.

제약:
- 결제 승인 로직은 수정하지 않는다.
- API 응답 스키마는 변경하지 않는다.
- UI 텍스트는 기존 i18n 구조를 따른다.
```

이 구조는 단순히 "환불 상태 배지 만들어줘"보다 훨씬 안정적입니다. 사람이 중간에 여러 번 개입해서 해야 할 확인을 루프 안에 넣었기 때문입니다.

## 7. 컨텐츠 제작에도 쓸 수 있다

영상에서는 이미지를 HTML Canvas와 JavaScript로 비슷하게 재현하는 예시가 나옵니다. 핵심은 한 번에 완성하려 하지 않고, 매 반복마다 결과를 렌더링하고 원본과 비교한 뒤 비평을 다음 수정에 반영하는 방식입니다.

글쓰기에도 비슷하게 적용할 수 있습니다.

```text
목표:
Loop Engineering 입문 글을 작성한다.

루프:
1. 초안을 작성한다.
2. 독자가 헷갈릴 용어를 표시한다.
3. 예시가 부족한 섹션을 보강한다.
4. 과장된 표현과 근거 없는 주장을 제거한다.
5. 제목, excerpt, 관련 글 링크를 점검한다.
6. 최종 글이 5분 안에 읽히는지 확인한다.
```

다만 컨텐츠는 정량 평가가 어렵습니다. "유사도 90%" 같은 명확한 기준이 없을 때는 반복 횟수 제한과 사람 리뷰 지점을 반드시 두는 편이 좋습니다.

## 8. 마치며

Loop Engineering은 완전히 새로운 마법이 아닙니다. 이미 우리가 하던 반복 개선을 더 높은 추상화로 올린 것입니다.

정리하면 다음과 같습니다.

1. **초기 프롬프트는 여전히 중요하다.**
2. **줄여야 하는 것은 반복마다 사람이 넣는 후속 프롬프트다.**
3. **루프는 행동, 관찰, 평가, 수정, 종료 조건으로 설계한다.**
4. **작은 작업에는 과할 수 있다.**
5. **큰 코드베이스, 반복 검증, 품질 개선 작업에서 특히 유용하다.**

트렌드를 따라가기보다 먼저 내가 반복해서 귀찮게 하고 있는 일이 무엇인지 보세요. "이거 또 확인해줘", "다시 고쳐줘", "테스트 돌리고 실패하면 수정해줘"를 반복하고 있다면, 그 지점이 Loop Engineering을 적용할 타이밍입니다.
