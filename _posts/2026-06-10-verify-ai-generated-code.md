## 1. 왜 검증이 별도의 기술인가

AI가 생성한 코드의 위험은 명백한 오류가 아닙니다. 컴파일도 되고 얼핏 동작하는데 **엣지 케이스에서만 틀리거나, 존재하지 않는 API를 그럴듯하게 호출하거나, 요구사항을 미묘하게 다르게 해석한** 코드입니다. 사람의 실수와 분포가 다르기 때문에, 사람 코드를 리뷰하던 습관만으로는 구멍이 생깁니다.

> 원칙은 한 줄입니다: **신뢰하되, 검증하라(Trust, but verify). 그리고 검증은 사람이 아니라 시스템이 하게 하라.**

---

## 2. 검증의 4단계

### 1단계 — 기계적 검증 (비용 0에 가깝게)

컴파일·린트·타입 체크·포매터는 에이전트가 작업 중에 스스로 돌리게 만듭니다. 설정 파일(AGENTS.md)에 "수정 후 반드시 `npm run lint && npm run typecheck` 통과"를 명시하면, 가장 값싼 오류는 사람에게 도달하기 전에 걸러집니다.

### 2단계 — 테스트 기반 검증 (핵심)

가장 강력한 검증 장치는 테스트입니다. 두 방향으로 씁니다.

- **기존 테스트 = 회귀 방지망**: 테스트 커버리지가 있는 코드베이스에서는 AI에게 훨씬 과감하게 위임할 수 있습니다. 커버리지가 없다면, AI 도입 1순위 작업은 기능 구현이 아니라 **기존 동작을 고정하는 테스트 작성**입니다.
- **테스트 먼저 합의**: 구현 전에 "이 동작을 검증하는 테스트부터 작성하라"고 시키고, 테스트가 요구사항을 제대로 반영하는지 **사람이 테스트를 리뷰**합니다. 테스트가 맞으면 구현은 통과 여부로 기계가 판정합니다.

주의: 같은 세션에서 구현과 테스트를 함께 시키면 **구현의 버그를 그대로 통과시키는 테스트**가 나올 수 있습니다. 테스트 리뷰가 구현 리뷰보다 중요해지는 이유입니다.

### 3단계 — 사람 리뷰 (판단이 필요한 곳에 집중)

1·2단계가 갖춰지면 사람은 기계가 못 잡는 것에 집중할 수 있습니다.

- 요구사항 해석이 맞는가 (가장 흔한 실패 지점)
- 아키텍처·컨벤션과 맞는가, 과도한 방어 코드·불필요한 추상화는 없는가
- 보안·트랜잭션·동시성 같은 도메인 판단

diff가 커서 리뷰가 불가능하다면 그것은 리뷰의 실패가 아니라 **작업 분할의 실패**입니다. 작은 단위로 다시 시키세요.

### 4단계 — 점진적 신뢰 조정

검증 강도는 고정이 아닙니다. 작업 유형별로 실적이 쌓이면 조정합니다.

| 작업 유형 | 초기 검증 | 신뢰 확보 후 |
|---|---|---|
| 문서·주석·테스트 작성 | 가벼운 리뷰 | 훑어보기 |
| 보일러플레이트·DTO | 전체 리뷰 | 샘플링 리뷰 |
| 비즈니스 로직 | 전체 리뷰 + 테스트 필수 | 전체 리뷰 유지 |
| 결제·보안·인프라 | 리뷰 2인 + 사람 게이트 | 완화하지 않음 |

---

## 3. 환각을 다루는 실전 요령

- **존재하지 않는 API**: 의존성 버전을 컨텍스트에 명시하고, "이 라이브러리 버전의 공식 문서에 있는 API만 사용하라"고 지시합니다. 빌드가 1차 방어선입니다.
- **자신만만한 오답**: "확신이 없는 부분은 확신이 없다고 표시하라"는 지시가 의외로 효과적입니다. 검증이 필요한 지점을 AI 스스로 표시하게 만드세요.
- **출처 요구**: 설계 결정이나 사실 주장에는 근거(코드 위치, 문서 링크)를 요구합니다. 근거를 못 대면 검증 대상입니다.

---

## 4. 마치며

AI 코딩의 생산성은 생성 속도가 아니라 **검증 처리량**에서 결정됩니다. 기계적 검증을 자동화하고, 테스트로 방어망을 깔고, 사람의 리뷰를 판단이 필요한 곳에 집중시키는 팀이 같은 모델로 몇 배의 차이를 만듭니다. 검증 체계는 어떤 도구를 쓰든 — Claude Code든 Codex든 Gemini CLI든 — 그대로 통하는 공통 자산입니다.
