## 1. 개요: 기존 RAG(검색)의 한계와 새로운 패러다임

우리가 흔히 사용하는 기존의 **RAG(Retrieval-Augmented Generation, 검색 증강 생성)** 방식은 근본적으로 '검색'에 의존합니다. 사용자가 질문할 때마다 관련 문서를 벡터 데이터베이스에서 검색해 AI에 프롬프트로 꽂아주고 답변을 생성한 뒤, 시스템은 모든 대화 컨텍스트를 잊어버립니다. 

안드레 카파시(Andrej Karpathy)는 이러한 방식이 비효율적이고 파편화되어 있다고 말합니다. 대신 그는 **지식을 프로그램 소스 코드처럼 관리하고, AI 에이전트를 통해 이를 체계적인 '위키(Wiki)'로 컴파일(Compile)하는 방식**을 제안했습니다. 이것이 바로 **LLM Wiki** 패러다임입니다.

---

## 2. LLM Wiki의 핵심 아키텍처

카파시는 이 구조를 소프트웨어 개발 환경에 빗대어 설명합니다.

> **"Obsidian은 IDE이고, AI 에이전트(Claude Code 등)는 프로그래머이며, Wiki 마크다운 파일들은 소스 코드(코드베이스)이다."**

- **소스 폴더 (Source Folder):** 정리가 안 된 오리지널 자료들(논문 PDF, 북마크 링크, 메모 스크랩 등). 이는 불변(Immutable) 상태로 유지되는 날 것의 정보입니다.
- **AI 에이전트 (Programmer):** 사람이 소스 폴더에 새로운 파일을 넣으면, 백그라운드에서 AI 에이전트가 이를 읽어 분석하고 핵심을 요약합니다.
- **위키 폴더 (Codebase):** 에이전트가 컴파일하여 만들어낸 상호 연결된 마크다운 파일들의 집합(`[[wikilinks]]` 구조). 사용자는 이 정돈된 위키를 편하게 읽고 탐색합니다.

---

## 3. RAG vs LLM Wiki 비교

| 구분 | 일반적인 RAG 환경 | LLM Wiki 워크플로우 |
| :---: | :--- | :--- |
| **정보 처리 방식** | 질문할 때마다 실시간으로 관련 문서를 검색 | 새로운 소스가 추가될 때마다 즉시 지식 베이스에 병합 |
| **지식의 형태** | 쪼개진 원본 텍스트 조각들 (Vector Chunks) | 사람이 읽기 쉽게 정돈되고 하이퍼링크된 마크다운 위키 |
| **에이전트의 역할** | 단순히 검색된 결과를 조합해 답변 생성 | 지식 간의 모순을 해결하고, 분류 체계를 동적 갱신 |
| **축적성** | 매 질문마다 일회성으로 소비 | 데이터가 쌓일수록 상호 연결 관계가 단단해짐 (지식의 복리 효과) |

---

## 4. 실전! LLM Wiki 구축 4단계 가이드

LLM Wiki를 시작하려면 거창한 서버나 데이터베이스가 필요 없습니다.

### Step 1. 필수 도구 준비하기 (환경 세팅)

1. **옵시디언(Obsidian) 설치**: 내 지식을 담아둘 뷰어이자 텍스트 에디터로 활용할 옵시디언을 PC에 설치합니다. 가볍고 오프라인으로 작동하며 데이터를 내 하드디스크에 영구 소유할 수 있습니다.
2. **옵시디언 웹 클리퍼(Web Clipper) 설치**: 크롬이나 엣지 등 브라우저 확장 프로그램에서 'Obsidian Web Clipper'를 설치합니다. 인터넷에서 발견한 유용한 기사나 유튜브 스크립트를 광고 없이 깔끔한 텍스트로 원클릭 수집할 수 있게 해줍니다.
3. **AI 에이전트 준비**: [Claude Code](https://github.com/anthropic/claude-code), 챗GPT 등 폴더 내의 파일을 읽고 쓸 수 있는 AI 도구를 준비합니다.

### Step 2. '3대 계층' 폴더 구조 만들기

옵시디언에 새로운 볼트(Vault, 보관소)를 하나 생성한 뒤, 시스템이 꼬이지 않도록 역할을 분리한 3가지 핵심 요소를 만듭니다.

* **① `raw` (또는 `sources`) 폴더 (원본 소스)**: 웹 클리퍼로 수집한 날것의 기사, PDF, 데이터 파일이 쌓이는 곳입니다. **[주의] 이곳의 파일은 절대 인간이 직접 수정해선 안 됩니다!** 오류 검증을 위한 영구적인 '진실의 기준점'으로 남겨두어야 합니다.
* **② `wiki` 폴더 (위키)**: AI가 `raw` 폴더의 원본을 읽고 스스로 요약, 분류, 연결하여 작성해 내는 마크다운 문서들이 저장되는 곳입니다.
* **③ `CLAUDE.md` (스키마/헌법 파일)**: 최상위 폴더에 이 파일을 생성합니다. AI에게 이 시스템을 어떻게 관리해야 하는지 알려주는 '운영 규칙서'입니다.

### Step 3. AI의 규칙(Schema)과 목적 설정하기

가장 중요한 단계입니다. 아무 정보나 무작정 모으면 쓰레기 데이터(Garbage In, Garbage Out)가 됩니다. `CLAUDE.md` 파일 안에 아래와 같은 **나의 핵심 맥락(목적)**과 **작업 규칙**을 적어 AI에게 주입해 주세요.

> **[CLAUDE.md 작성 예시]**
> - **나의 목적**: "나는 주식 투자와 반도체 산업 트렌드를 공부하고 있어. 최종적으로 투자 의사결정을 돕는 분석 대시보드를 구축하는 게 목적이야."
> - **작업 규칙 (Ingest)**: "새로운 자료가 `raw` 폴더에 들어오면, 요약 페이지를 만들고 기존 개념 페이지와 연결해 줘. `index.md`(목차 파일)도 잊지 말고 갱신해 줘."
> - **작업 규칙 (Query)**: "내가 질문을 던져서 나온 훌륭한 답변이나 비교 분석표는 대화창에만 두지 말고, `wiki` 폴더 안에 새로운 페이지로 꼭 저장해 줘."

### Step 4. 실전 활용법: 3대 핵심 루프 (Ingest - Query - Lint)

이제 매일 반복하며 지식을 눈덩이처럼 불려나갈 차례입니다.

1. **지식 입력 (Ingest)**: 웹 서핑 중 좋은 글을 발견하면 웹 클리퍼로 `raw` 폴더에 수집합니다. 그리고 AI에게 "새로 추가된 문서를 위키에 반영해 줘"라고 지시합니다. 이때 *"이 문서는 향후 AI 트렌드를 예측하기 위해 수집했어"*처럼 나만의 수집 목적과 코멘트를 덧붙여주면 AI가 훨씬 퀄리티 높은 문서를 만들어냅니다. 단일 문서 하나만 넣어도 AI는 관련된 기존 페이지 10~15개를 알아서 동시다발적으로 업데이트합니다.
2. **지식 탐구 (Query)**: 지식이 어느 정도 쌓이면 AI에게 질문을 던집니다. AI는 흩어진 위키 문서들을 엮어 깊이 있는 답변을 내놓습니다. 여기서 가장 중요한 점은, **이 탐구의 결과물(답변) 자체를 위키의 새로운 지식 노트로 다시 저장**하라고 지시하는 것입니다.
3. **건강 검진 (Lint)**: 가끔 AI에게 "위키 상태를 점검해 줘"라고 요청합니다. 과거에는 호재라고 했던 정보가 오늘 수집된 뉴스에선 악재로 바뀌어 모순이 발생할 수 있습니다. AI는 논리적 충돌, 끊어진 링크, 낡은 정보를 스스로 찾아내어 타임라인에 맞게 역사를 재정리하고 시스템을 자가 치유합니다.

## 5. 마치며: '읽기 전용' 인간과 '쓰기 전용' AI

지식 관리의 패러다임은 **"내가 직접 쓰고 정리하던 시대"**에서 **"AI가 고도로 정리해 놓은 결과물을 내가 검토하고 소비하는 시대"**로 전환되고 있습니다. 

안드레 카파시의 LLM Wiki는 단순 검색 도구에 의존하던 AI 활용법을 **능동적인 개인 지식 컴파일러**로 한 단계 격상시키는 실전 디자인 패턴입니다. 늘어나는 북마크와 읽지 않는 PDF 더미 속에서 지식의 부채를 겪고 있다면, 오늘 당장 나만의 LLM Wiki 폴더를 구성해 보시기 바랍니다.
