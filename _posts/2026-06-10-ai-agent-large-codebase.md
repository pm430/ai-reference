## 1. 왜 대규모 코드베이스는 다른가

토이 프로젝트에서 잘 되던 AI 에이전트가 회사 코드베이스에서는 헤매는 이유는 명확합니다.

1. **컨텍스트 한계**: 1M 토큰 모델이라도 수십만 줄 전체를 "이해"한 상태로 작업할 수는 없습니다. 모델은 결국 일부만 읽고 추론합니다.
2. **암묵지**: "이 모듈은 건드리면 안 된다", "이 코드는 곧 폐기된다" 같은 지식은 코드에 없습니다.
3. **일관성 비용**: 같은 패턴이 300곳에 있을 때, 한 곳만 다르게 고치면 새로운 부채가 됩니다.

> 대규모 코드베이스에서 AI 도입의 성패는 모델 성능이 아니라 **작업 범위를 얼마나 잘 잘라주느냐**에서 갈립니다.

---

## 2. 도입 순서 — 위험이 낮은 것부터

처음부터 "이 기능 구현해줘"로 시작하면 실패합니다. 신뢰를 쌓는 순서가 있습니다.

| 단계 | 작업 | 위험도 |
|---|---|---|
| 1 | **읽기 전용**: 코드 설명, 호출 흐름 추적, 영향도 분석 | 없음 |
| 2 | **문서화**: README·AGENTS.md 생성, 모듈별 요약 작성 | 없음 |
| 3 | **테스트 추가**: 기존 동작을 고정하는 특성화 테스트(characterization test) | 낮음 |
| 4 | **소규모 수정**: 단일 모듈 내 버그 수정, 리팩토링 | 중간 |
| 5 | **기능 구현**: 테스트와 리뷰 체계가 갖춰진 뒤에만 | 높음 |

특히 1~2단계는 즉시 효과가 큽니다. 10년 묵은 시스템의 "이 배치가 왜 도는지 아는 사람이 퇴사한" 상황에서, 호출 흐름을 추적해 문서로 남기는 일은 AI가 사람보다 빠르고 지치지 않습니다.

---

## 3. 범위 설정의 기술

### 모듈 단위로 자르기

"전체 코드베이스 리팩토링"이 아니라 **"`order-service` 모듈의 `OrderValidator`와 그 테스트"**처럼 경계를 명시합니다. 모노레포라면 작업 디렉터리 자체를 하위 패키지로 한정하는 것도 효과적입니다.

### 계층별 AGENTS.md / CLAUDE.md

대규모 프로젝트에서는 루트 설정 파일 하나로 부족합니다. 모듈별로 설정 파일을 두면 에이전트가 해당 디렉터리에서 작업할 때 그 모듈의 규칙을 읽습니다.

```text
repo/
├── AGENTS.md              # 전사 공통: 빌드, 금지 영역
├── payment/
│   └── AGENTS.md          # "이 모듈은 PCI 감사 대상, 로깅 규칙 엄수"
└── legacy-batch/
    └── AGENTS.md          # "폐기 예정. 버그 수정만, 기능 추가 금지"
```

### 컨텍스트에 "지도"를 먼저 주기

작업 전에 모듈 구조 요약, 핵심 진입점, 데이터 흐름을 담은 짧은 아키텍처 문서를 읽히면 탐색 비용이 크게 줄어듭니다. 무엇을 컨텍스트에 담을지의 일반론은 [Context Engineering]({% post_url 2026-06-08-context-engineering %})에서 다룹니다.

## 4. 실패하는 패턴

- **한 번에 큰 PR**: 수천 줄 diff는 사람이 리뷰할 수 없고, 리뷰할 수 없는 변경은 머지할 수 없습니다. 작은 PR 여러 개로 강제하세요.
- **테스트 없는 리팩토링**: 기존 동작을 고정하는 테스트 없이 레거시를 고치게 하면, 무엇이 깨졌는지 알 방법이 없습니다.
- **암묵지를 말로만 전달**: 세션이 끝나면 사라집니다. 대화에서 알려준 규칙은 반드시 AGENTS.md에 적어 영속화하세요.
- **전 팀 동시 도입**: 파일럿 모듈 하나에서 워크플로우(브랜치 규칙, 리뷰 기준)를 정착시킨 뒤 확산하는 편이 빠릅니다.

---

## 5. 마치며

대규모 코드베이스에서 AI 에이전트는 "코드를 대신 짜주는 도구"이기 전에 **"코드베이스를 읽고 설명해주는 도구"**로 먼저 가치를 냅니다. 읽기 → 문서화 → 테스트 → 수정의 순서를 지키면, 기간계 시스템에서도 안전하게 생산성을 끌어올릴 수 있습니다.
