## 1. Context Engineering이란?

2025년 말부터 AI 업계에서 **Context Engineering** 이라는 용어가 빠르게 자리 잡았습니다. Anthropic의 문서 *"Effective context engineering for AI agents"* 가 대표적이고, 한국어로는 '맥락 공학' 으로 옮기기도 합니다.

Prompt Engineering이 *"어떻게 질문하는가"* 에 집중한다면, Context Engineering은 **"어떤 정보를 어떤 순서로 컨텍스트 창에 담을 것인가"** 에 집중합니다. AI가 한 번에 볼 수 있는 토큰 수는 한계가 있으므로, 그 안에 **무엇을 넣고 무엇을 빼느냐** 가 결과물을 좌우합니다.

---

## 2. Why: 왜 Prompt Engineering만으로 부족한가

프롬프트는 *질문 그 자체* 에 집중하지만, 실제 LLM 호출은 그보다 훨씬 많은 정보를 컨텍스트에 담습니다.

- 시스템 프롬프트 (역할, 규칙, 예시)
- 도구 사용 기록 (Function calling 결과)
- 검색/검색 결과 (RAG로 가져온 문서)
- 이전 대화 기록
- 사용자 입력
- 응답 형식 정의

이 모든 조각을 *어떤 우선순위로, 어떤 길이로, 어떤 포맷으로* 담을지가 결과 품질을 결정합니다. 단순한 "잘 묻는 법" 만으로는 통제할 수 없는 영역입니다.

---

## 3. Context Engineering의 4가지 핵심 결정

### 3.1 무엇을 담을 것인가 (Selection)

가장 큰 실수: **모든 것을 다 담는 것**. 컨텍스트가 길어질수록 모델의 주의가 분산되고, 비용·속도·정확도 모두 나빠집니다.

**원칙**: *지금 답하려는 질문과 직접 관련된 정보만* 담습니다.

```text
❌ Bad: 코드베이스 전체 + 문서 전체 + 이전 대화 + 사용자 질문
✅ Good: 관련 파일 3개 + 검색된 문서 단락 2개 + 시스템 규칙 + 사용자 질문
```

**실무 팁**:
- 도구 결과를 *요약해서* 넣을지, *원본 그대로* 넣을지 결정합니다. 일반적으로 1,000줄짜리 JSON 원본보다, 핵심 필드만 추출한 50줄 요약이 효과적입니다.
- 컨텍스트 한도(예: 200K)의 **70%만 사용**하는 것을 목표로 합니다. 여유분은 모델이 *응답 생성*에 써야 합니다.

### 3.2 어떤 순서로 배치할 것인가 (Ordering)

컨텍스트 안의 *위치*도 결과에 영향을 줍니다. **가장 중요한 정보는 앞쪽과 끝쪽**에 배치하는 것이 일반적으로 효과적입니다 (Primacy/Recency 효과).

```text
[System]                 ← 역할, 핵심 규칙 (가장 중요)
[가장 관련성 높은 도구/문서 결과]   ← 두 번째
[이전 대화 요약]          ← 맥락
[사용자 입력]            ← 질문
[출력 형식 정의]         ← 마지막
```

LLM마다 차이는 있지만, 이 순서는 대부분의 벤치마크에서 안정적입니다.

### 3.3 어떤 포맷으로 표현할 것인가 (Format)

같은 정보라도 *어떻게 표현하느냐*에 따라 모델의 활용도가 달라집니다.

| 정보 | ❌ 나쁜 포맷 | ✅ 좋은 포맷 |
| --- | --- | --- |
| 도구 사용 결과 | 한 줄 요약 | `<tool_result>...</tool_result>` XML |
| 검색된 문서 | 평문 덤프 | 출처·페이지·요약 메타데이터 포함 |
| 이전 대화 | 메시지 통째 복사 | 핵심 결정사항만 발췌 |
| 사용자 입력 | 자연어 | 명시적 슬롯 (`{question}`, `{context}`) |

**XML 태그**는 Anthropic이 적극 권장하는 패턴입니다. 모델이 섹션 경계를 더 잘 인식해, *어떤 정보를 응답에 활용해야 하는지* 판단이 명확해집니다.

```xml
<instructions>
사용자의 질문에 한국어로 답하라. 코드 예시는 Python을 사용한다.
</instructions>

<context>
[검색된 문서 1]
URL: https://docs.example.com/api/orders
요약: 주문 API는 POST /api/orders 로 호출한다.
</context>

<question>
결제 실패 시 멱등성 보장은 어떻게 처리하나?
</question>
```

### 3.4 무엇을 잊게 할 것인가 (Compaction)

긴 대화에서 컨텍스트가 한도를 초과하면, *잊는 것*도 능동적으로 관리해야 합니다. 이것이 **Compaction** 입니다.

```text
[원본 대화 — 50,000 토큰]
   ↓ 압축
[요약 — 2,000 토큰]
- 사용자 의도: 결제 시스템의 멱등성 검증
- 핵심 결정: orderId 기반 idempotency key 사용
- 보류 이슈: 동시 요청 시 race condition 가능성
- 다음 단계: 격리 테스트 작성
```

**전략**:
- **요약 기반 압축**: 5~10턴 이전 대화는 요약으로 치환
- **트리밍**: 결정에 직접 기여하지 않은 메시지(인사, 부수적 질문)는 제거
- **참조 유지**: 핵심 코드 블록, 명세 문서는 그대로 두고 주변 대화만 압축

---

## 4. 흔한 안티패턴

- **컨텍스트를 무한히 늘리기** — 비용이 선형으로 증가하는 데 비해, 품질 향상은 로그 곡선처럼 느려집니다.
- **사용자 메시지에 모든 컨텍스트를 우겨넣기** — 시스템 프롬프트의 역할과 메시지의 역할이 섞이면 모델이 우선순위를 혼동합니다.
- **도구 결과를 잘라 붙이지 않기** — `"DB 조회 결과: [50,000자 JSON]"` 처럼 통째로 넣으면 모델이 무시합니다. 핵심만 추출해 주세요.
- **Compaction 없이 계속 이어가기** — 50턴이 지난 대화에서 모델은 앞부분을 *잊습니다*. 명시적 요약이 필요합니다.

---

## 5. RAG와의 관계

RAG(Retrieval-Augmented Generation)는 **Context Engineering의 한 사례**로 볼 수 있습니다. 검색을 통해 *관련 정보만* 컨텍스트에 동적으로 채워 넣는 것이 본질이죠.

하지만 RAG만으로 충분하지 않습니다. 다음이 함께 가야 합니다.

- **Re-ranking**: 검색 결과 상위 K개를 모델로 다시 정렬해 관련도 높은 것이 컨텍스트 앞쪽에 오게 함
- **Chunking 전략**: 문서를 *어디서 자를지*가 검색 품질을 좌우 (heading 단위, 의미 단위, 고정 길이)
- **Citation**: 어떤 출처에서 가져왔는지 컨텍스트에 명시 → 모델이 출처 인용 가능

이 모든 것이 결국 "어떤 정보를 컨텍스트에 담을 것인가" 의 변형입니다.

---

## 6. 실전 워크플로우 — Claude Code의 Skills

이 사이트의 다른 글에서 다루는 **Skills, Subagents, Commands** 도 Context Engineering의 도구입니다.

- **Skills**: 자주 쓰는 도메인 지침을 *재사용 가능한 모듈*로 만들어 컨텍스트에 동적 주입
- **Subagents**: 무거운 작업(긴 검색, 코드 분석)을 *별도 컨텍스트에서* 실행하고 요약만 메인에 전달
- **Commands**: 자주 쓰는 지시 패턴을 슬래시 커맨드로 묶어 두어 *입력 단계*를 줄임

핵심 통찰: **컨텍스트는 비싼 자원**입니다. 매번 새로 쓰지 말고, 모듈화하고 재사용하고, 무거운 작업은 격리하세요.

---

## 7. 한 단계 더 — Compaction 자동화

장기 실행되는 에이전트(예: 8시간 자율 코딩 세션)는 *자동으로* 컨텍스트를 압축하는 메커니즘이 필수입니다. Claude Code의 hooks가 이를 담당합니다.

```text
[Hook 예시 — 컨텍스트 80% 도달 시 자동 요약]
1. 현재 컨텍스트 길이 확인
2. 80% 초과 시 가장 오래된 20%를 LLM으로 요약
3. 원본은 외부 저장소에 보존 (필요 시 복원 가능)
4. 요약본을 컨텍스트에 삽입
```

이 패턴은 Anthropic, LangChain, LlamaIndex 등 대부분의 에이전트 프레임워크가 지원합니다. 직접 구현하지 말고 검증된 라이브러리의 *컴팩션 미들웨어*를 사용하세요.

---

## 8. 마치며

Context Engineering은 **"AI에게 무엇을 주고 무엇을 뺄 것인가"** 라는 단순한 질문에서 출발합니다. 그 답을 잘 내리면, 같은 모델을 써도 결과 품질이 크게 달라집니다.

다음 네 가지를 기억하세요.

1. **Selection**: 관련 정보만, 컨텍스트 한도의 70% 이내
2. **Ordering**: 중요한 정보는 앞/뒤, 도구 결과는 그 다음
3. **Format**: XML 태그로 섹션 구분, 메타데이터 명시
4. **Compaction**: 10턴 이전은 요약으로, 무거운 작업은 격리

Prompt Engineering이 *질문의 기술*이라면, Context Engineering은 *환경을 설계하는 기술*입니다. 두 가지를 함께 익혀야 비로소 AI를 *잘* 쓸 수 있습니다.
