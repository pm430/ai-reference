## 1. 디버깅은 왜 어려운가

코딩만큼 디버깅이 힘든 작업도 드뭅니다. 특히 다음 세 가지 상황에서는 혼자 머리를 싸매도 답이 안 나올 때가 많습니다.

- **재현이 안 되는 버그**: 운영 환경에서만 발생, 사용자가 알려준 환경 정보만으로는 부족
- **알 수 없는 실패**: 스택트레이스가 비즈니스 코드 깊숙이 묻혀 있음
- **타인의 코드**: 3년 전 동료가 짜고 아무도 손대지 않은 모듈

이런 상황의 공통점은 **"내가 보는 정보로는 답이 없다"** 는 점입니다. 그래서 외부 페르소나의 도움이 절실합니다.

---

## 2. Why: AI는 디버깅에 어떻게 도움이 되는가

AI는 코드를 *고치는* 것보다 **생각을 정리하게 만드는** 데 훨씬 큰 가치가 있습니다. 다음 세 가지 방식으로 작동합니다.

1. **가설 생성**: "이 증상이면 보통 세 가지 원인이 있다" 같은 체크리스트를 빠르게 던져 줍니다. 사람은 흔들릴 때 떠올리는 가설이 한두 개에 그치지만, 모델은 수십 개를 즉시 제시할 수 있습니다.
2. **격리 실험 설계**: "이 가설을 확인하려면 어떤 로그를 찍어야 하는가"를 구체화해 줍니다.
3. **타인의 관점**: 본인이 작성한 코드일수록 사각지대가 큽니다. 모델은 "이 코드를 처음 보는 사람"의 시선을 빌려 줍니다.

다만 결정적 한계가 있습니다. **모델은 직접 실행하거나 로그를 읽을 수 없습니다.** 사용자가 정보를 잘 전달하지 않으면 그만큼 효과가 떨어집니다.

---

## 3. How: AI와 함께하는 5단계 디버깅

### Step 1. 재현 정보 정리 — 컨텍스트를 잘 던져라

AI에게 한 문장으로 "버그 있어요" 라고 던지지 마세요. 다음 5가지를 한 번에 정리해 보냅니다.

```text
환경: Python 3.11, FastAPI 0.110, PostgreSQL 15 (production only, staging OK)
증상: POST /api/orders 가 1% 확률로 500을 반환. 응답 본문은 {"detail":"Internal Server Error"} 만 있음.
시점: 2026-05-20 이후로 발생. 그 전에는 정상이었음.
변경 이력: 같은 날 feat/order-v2 브랜치를 배포함. (diff: [paste])
로그: 스택트레이스는 첨부함. 단, 마지막 프레임은 sqlalchemy.exc.OperationalError 로 끝남.

가장 가능성이 높은 원인이 무엇인지 우선순위 3개로 정리해줘.
```

**원칙**: 환경/증상/시점/변경/로그. 이 다섯 가지를 빠짐없이 적는 것만으로 AI 답변의 질이 두 단계는 올라갑니다.

---

### Step 2. 가설 좁히기 — "왜"를 한 단계씩

AI가 제시한 가설을 그대로 믿지 말고, **실험으로 한 번씩 좁혀 갑니다**.

```text
가설 1: 커넥션 풀 고갈
→ 확인 방법: 운영 DB의 active connections 카운트, pgbouncer의 wait_time
→ 결과: 평균 12, 최대 80 (pool size 100). 가능성 있음.

가설 2: feat/order-v2 의 트랜잭션 경계 변경
→ 확인 방법: feat/order-v2 PR에서 with session.begin() 호출 위치 확인
→ 결과: 함수 중간에서 session.close() 가 명시적으로 호출되고 있었음. 가능성 매우 높음.

가설 3: DB 락 경합
→ 확인 방법: pg_stat_activity 에서 lock 대기 시간 확인
→ 결과: 평균 대기 5ms 이하. 가능성 낮음.

다음으로 가설 2 를 검증하기 위해 어떤 로그를 어디에 찍으면 될까?
```

이렇게 **대화 기록이 누적되면**, 같은 세션을 나중에 다시 열어 다른 동료에게 그대로 공유할 수 있습니다. "이 버그 추적한 로그" 가 사내 위키보다 더 정확합니다.

---

### Step 3. 재현 코드 작성 — AI에게 격리 테스트를 부탁

```text
위 가설 2(session.close() 명시 호출) 가 맞는지 격리 테스트로 확인하고 싶어.
다음 조건을 만족하는 pytest 코드를 작성해줘.

- in-memory SQLite 로 FastAPI 앱 부팅
- 동일 세션에서 session.close() 호출 후 추가 쿼리 시도
- 500 응답이 나와야 함

그리고 같은 코드에서 session.close() 호출을 제거했을 때 응답이 200 으로 바뀌는지도 함께 검증해줘.
```

이렇게 하면 **"원인을 안다" 에서 "원인을 증명한다"** 로 한 단계 올라갑니다. AI가 만든 격리 테스트가 통과하면, 패치는 거의 확정입니다.

---

### Step 4. 패치 — 최소 단위로

원인이 확정되면 패치는 신중하게.

```text
원인: feat/order-v2 에서 FastAPI dependency 함수 안에서 session.close() 를 명시 호출.
세션은 dependency 가 관리하므로 명시 close 가 들어가면 후속 요청에서 세션이 닫힌 채 재사용됨.

수정안: session.close() 호출 제거. dependency 가 알아서 정리하게 둠.

이 변경으로 사이드 이펙트가 있을 수 있는 시나리오 3개만 알려줘.
```

**"사이드 이펙트 시나리오 알려줘"** 한 줄이 중요합니다. AI가 잊고 있는 엣지 케이스를 짚어주며, 본인이 검토할 체크리스트가 만들어집니다.

---

### Step 5. 회귀 방지 — 같은 종류 버그가 다시 안 나오게

```text
이번 버그의 근본 원인이 "dependency 가 관리하는 세션을 코드에서 직접 만짐" 이었어.
이런 종류의 실수를 CI 에서 잡으려면 어떤 검사를 추가하면 좋을까?

- 정적 분석으로 잡을 수 있는 방법
- 런타임 가드
- 코드 리뷰 체크리스트
세 가지를 제안해줘.
```

**이 단계가 없으면 같은 버그가 6개월 뒤에 다시 나옵니다.** 코드 한 줄을 고친 것보다, 같은 종류의 버그가 다시 안 나오게 *구조를* 바꾼 것이 더 큰 수확입니다.

---

## 4. AI가 틀렸을 때 — 환각을 역이용하는 법

AI는 이 작업에서도 잘 틀립니다. **틀리는 패턴은 거의 정형화**되어 있습니다.

- **너무 그럴듯한 가설을 단정적으로 제시** — "이거 원인 맞다" 식으로 답하면 의심해 봅니다.
- **없는 API / 없는 옵션을 지어냄** — "이 함수의 두 번째 인자에 strict=True 를 넣어봐" 같은 답이 나올 수 있는데, 실제로는 그런 인자가 없을 수 있습니다.
- **레거시/구버전의 동작을 모름** — 한참 업데이트되지 않은 라이브러리의 정확한 시그니처를 종종 틀립니다.

이럴 때의 대처법:

| 증상 | 대처 |
| --- | --- |
| 그럴듯한 단정 | "이 가설을 부정할 수 있는 반례를 2개 제시해줘" |
| 없는 API·옵션 | "이 옵션이 존재하는 공식 문서 URL 을 알려줘" |
| 구버전 지식 | "v0.110 기준이야. 그 이전 버전 정보는 무시해" |

**환각을 *탐지*하는 가장 빠른 방법은 "그럴듯한 답을 의심 없이 받아들이지 않는 습관"** 입니다.

---

## 5. 흔한 안티패턴

- **"이 코드 왜 안 돼?"** 만 던지고 끝 — 컨텍스트가 부족하면 AI 도 답할 수가 없습니다.
- **AI 답변을 그대로 패치** — 한 번 더 사람이 읽고, 격리 테스트로 확인합니다.
- **대화 기록 없이 진행** — 디버깅 세션이 길어지면 맥락이 사라집니다. 핵심 발견은 텍스트 파일로 추출해 두세요.
- **처음부터 AI에게 패치 요청** — 가설 검증 → 격리 테스트 → 패치의 순서를 거꾸로 하면, "왜 이 패치인지"를 설명할 수 없게 됩니다.

---

## 6. 한 단계 더 — 로그를 자동으로 요약하는 환경

운영 환경에서 매번 1% 확률 버그 로그를 사람이 읽는 건 비효율적입니다.

- **구조화 로그(JSON)**: AI 가 파싱하기 좋은 형태로 남기면, 사후 분석이 10배 빨라집니다.
- **에러 발생 시 자동 컨텍스트 첨부**: traceback 외에 최근 N개 요청, 사용 환경, DB 상태를 같이 저장.
- **ReAct 패턴의 로그 분석 에이전트**: Smithery 같은 MCP 레지스트리에서 "Sentry/ElasticSearch" 같은 도구를 등록해 두면, AI가 직접 로그를 조회해 가설을 세울 수 있습니다.

이 도구들을 깔아 두면, 다음에 비슷한 버그가 터졌을 때 **본인이 출근하기도 전에 가설 리스트가 메일함에 들어와 있는** 경험을 할 수 있습니다.

---

## 7. 마치며

AI는 디버깅에서 *정답을 맞히는 도구*가 아니라, **생각을 정리하고 가설을 빠르게 좁히는 동료**입니다. 컨텍스트를 잘 던지고, 가설을 좁히고, 격리 테스트로 확인하고, 패치는 신중하게 — 이 순서를 지키면 AI 없이 한 시간 걸리던 작업이 10분으로 줄어듭니다.

핵심 한 줄: **AI를 "원인 추측기"로 쓰지 말고, "가설 정리와 격리 실험 설계"에 쓰세요.** 그 역할에 가장 잘 맞습니다.
