## CLAUDE.md가 뭔가

`CLAUDE.md`는 Claude Code가 **매 세션 시작 때 자동으로 읽는** 프로젝트 메모리 파일입니다. 프로젝트 루트에 두면, 매번 "이 프로젝트는 이렇게 빌드하고, 이런 규칙을 지켜"라고 설명할 필요 없이 Claude가 알아서 그 맥락을 안고 시작합니다.

즉 CLAUDE.md는 **반복 설명을 한 번만 적어두는 곳**입니다. 잘 쓰면 매 대화의 품질이 올라가고, 잘못 쓰면 컨텍스트만 잡아먹습니다.

---

## 어디에 두나

CLAUDE.md는 위치에 따라 적용 범위가 다릅니다.

| 위치 | 범위 | 용도 |
|---|---|---|
| 프로젝트 루트 `./CLAUDE.md` | 그 저장소 | 빌드·테스트 명령, 코딩 규칙, 구조 |
| 홈 `~/.claude/CLAUDE.md` | 모든 프로젝트 | 개인 선호(말투, 공통 워크플로우) |
| 하위 디렉터리 `src/foo/CLAUDE.md` | 그 폴더 | 특정 모듈에만 해당하는 규칙 |

프로젝트 루트 것이 핵심입니다. 팀과 공유하려면 저장소에 커밋하고, 개인 메모는 홈에 둡니다.

## 무엇을 담나

> 좋은 CLAUDE.md는 "Claude가 매번 틀리거나 매번 물어보게 되는 것"을 미리 적어둔 것입니다.

담으면 효과 큰 것:

- **빌드·테스트·lint 명령** — `npm test`, `bundle exec jekyll build` 처럼 정확한 커맨드
- **코딩 규칙** — 네이밍, 포맷, 선호하는 패턴/금지 패턴
- **프로젝트 구조** — 주요 디렉터리가 무슨 역할인지 한 줄씩
- **함정(gotchas)** — "이 폴더는 자동 생성이니 손대지 마", "배포는 main 푸시로 트리거됨" 같은 비자명한 사실

`@경로/파일` 문법으로 다른 파일을 불러올 수도 있어, 긴 규칙은 별도 파일로 빼고 참조만 걸 수 있습니다.

## 무엇을 빼나

CLAUDE.md는 매 세션 컨텍스트를 소비합니다. 그래서 **짧을수록 좋습니다.**

- **장황한 설명** — 문단으로 풀어 쓰지 말고 목록·명령으로 압축
- **자주 바뀌는 정보** — 진행 중인 작업 상태, 임시 TODO는 금방 거짓말이 됨
- **비밀** — API 키·토큰은 절대 넣지 않음 (커밋되면 유출)
- **코드가 이미 말해주는 것** — 파일 구조나 함수 시그니처를 받아 적지 말 것

200줄이 넘어가면 십중팔구 과합니다. "이게 정말 매 세션 필요한가?"를 기준으로 덜어내세요.

## 좋은 CLAUDE.md의 모습

```markdown
# Project: ai-reference (Jekyll 블로그)

## 명령
- 빌드: bundle exec jekyll build
- 로컬 서버: bundle exec jekyll serve

## 규칙
- 글은 _posts/YYYY-MM-DD-slug.md, 한국어로 작성
- 카테고리는 _config.yml의 6개 중 하나만 사용
- 내부 링크는 {% raw %}{% post_url %}{% endraw %}, 대상 존재 확인 필수

## 함정
- GitHub Pages는 future: false — 미래 날짜 글은 건너뜀
- main 푸시가 곧 배포. 푸시 전 빌드 점검
```

핵심은 **구체적인 명령과 비자명한 함정**입니다. "좋은 코드를 작성하라" 같은 추상적 지시는 빼세요.

## 만들고 관리하기

- `/init` 명령으로 Claude가 코드베이스를 훑어 초안을 만들어줍니다. 그걸 다듬는 게 가장 빠릅니다.
- 대화 중 `#`로 시작하는 메모를 남기면 CLAUDE.md에 바로 추가됩니다.
- 규칙은 **이유와 함께** 적으면 Claude가 더 잘 따릅니다. "X 하지 마"보다 "X는 자동 생성되니 직접 고치면 덮어써짐"이 낫습니다.
