프롬프트 엔지니어링은 AI 모델에 보낼 텍스트의 구조를 설계해 최적의 결과물을 유도하는 소프트웨어 디자인 기법입니다. 이번 포스트에서는 다양한 LLM 애플리케이션을 개발할 때 모델의 성공률을 극적으로 향상시키는 3대 핵심 프롬프트 패턴과 작성 전략을 소개합니다.

---

## 1. 퓨샷 프롬프팅 (Few-Shot Prompting)

모델에게 원하는 출력 형식이나 스타일을 이해시키기 위해 몇 개의 **예시(Example)**를 프롬프트에 직접 포함시키는 기법입니다.

- **어떤 때 쓰나요?**
  - 원하는 JSON 구조나 특수한 포맷이 있을 때
  - 단어 선택이나 출력 톤앤매너를 일관되게 규정하고 싶을 때
- **잘못된 예시 (Zero-Shot):**
  > 사용자의 문장을 감정 분석해서 JSON으로 변환해줘. 문장: "이 가격에 이 퀄리티라니 진짜 대박이다."
- **좋은 예시 (Few-Shot):**
  ```markdown
  사용자 문장을 감정 분석하여 아래 포맷으로 변환해 주세요.

  [예시 1]
  문장: "배송이 너무 느려서 짜증 나네요."
  출력: { "sentiment": "negative", "score": 0.9, "reason": "배송 지연에 대한 불만족" }

  [예시 2]
  문장: "기대 이상으로 마감이 깔끔해요."
  출력: { "sentiment": "positive", "score": 0.85, "reason": "제품 마감 상태에 대한 만족" }

  [예시 3]
  문장: "이 가격에 이 퀄리티라니 진짜 대박이다."
  출력:
  ```

## 2. 생각의 사슬 (Chain-of-Thought, CoT)

LLM에게 답을 즉시 내기 전에 **단계적인 추론 과정**을 먼저 서술하도록 강제하는 기법입니다.

- **어떤 때 쓰나요?**
  - 복잡한 논리 연산, 수학 문제 해결, 코드 버그 분석
  - 모델이 성급한 결론을 도출해 오답을 내는 것을 예방할 때
- **핵심 작성 팁:**
  프롬프트 마지막 부분에 `"천천히 생각해서 단계적으로 해결해줘(Let's think step by step)"` 문구를 적거나, 생각 단계를 JSON의 `thinking` 필드에 나누어 작성하게 함으로써 오답률을 크게 낮출 수 있습니다.

---

## 3. 역할 정의 및 시스템 프롬프트 (Role-Playing & System Prompt)

LLM의 컨텍스트를 한정 짓기 위해 모델의 **페르소나**와 **동작 규칙**을 시스템 단에 단단히 주입하는 기법입니다.

- **어떤 때 쓰나요?**
  - AI 비서가 특정 비즈니스 도메인을 벗어나 엉뚱한 답변을 하는 것을 막을 때
  - 특정 프로그래밍 언어나 스타일 가이드를 강제할 때
- **시스템 프롬프트 예시:**
  ```markdown
  당신은 10년 경력의 시니어 React 개발자이자 모범적인 가이드를 제공하는 테크니컬 라이터입니다.
  아래 규칙을 반드시 준수하세요:
  1. 답변에 사용하는 모든 예제 코드는 TypeScript로 작성하며, 타입 정의를 생략하지 마세요.
  2. 성능 최적화(React.memo, useMemo 등) 관점을 함께 언급하세요.
  3. 한국어로 친절하고 정중하게 대답하되 불필요한 사담은 생략하세요.
  ```

## 개발자를 위한 프롬프트 작성 3계명

1. **지시사항은 명확하고 구체적으로 적으세요.**
   - ❌ "자바스크립트 코드 최적화해줘"
   - ⭕ "아래 JavaScript 함수에서 중첩 루프를 제거하고 시간 복잡도를 O(N)으로 개선하는 효율적인 로직을 작성해줘"
2. **출력 형태의 한계를 경계(Boundary)로 명시하세요.**
   - ❌ "다른 말은 하지 마"
   - ⭕ "답변에는 마크다운 코드 블록 이외의 추가 설명이나 텍스트를 포함하지 마세요. 코드 블록만 즉시 반환하세요."
3. **가장 복잡한 논리를 먼저 예시로 제공하세요.**
   - 퓨샷을 넣을 때 쉬운 예시보다, 실제 입력될 데이터 중 가장 복잡하고 예외적인 데이터의 예시를 주어야 예외 상황에서도 모델이 잘 대처합니다.
