## 1. 결제 도메인은 무엇이 다른가

일반 서비스 코드와 달리 결제·정산 시스템은 다음 제약 위에서 움직입니다.

- **되돌릴 수 없다**: 잘못 나간 돈은 "롤백"이 아니라 "환불·정산 보정"이라는 비싼 프로세스가 됩니다.
- **감사와 규제**: PCI DSS, 전자금융감독규정 등 — 코드 변경 자체가 감사 추적 대상입니다.
- **불변식이 생명**: 멱등성(같은 요청 두 번 = 결제 한 번), 원장 대사(원천 데이터와 장부 일치), 금액 계산의 정확성.

> 고위험 도메인의 원칙은 한 줄입니다: **AI의 산출물 속도는 취하되, 의사결정과 최종 책임은 시스템적으로 사람에게 묶어둔다.**

---

## 2. 맡길 일 vs 맡기면 안 되는 일

| 맡겨도 되는 것 (사람 리뷰 전제) | 맡기면 안 되는 것 |
|---|---|
| 멱등성·이중결제 시나리오의 **테스트 코드 작성** | 결제 승인·취소의 **핵심 분기 로직 단독 수정** |
| PG사 연동 스펙 문서 읽고 **요청/응답 DTO 생성** | **운영 DB·운영 설정** 접근이 필요한 모든 작업 |
| 장애 로그·대사 불일치 **원인 분석 보조** | 금액 계산식 변경을 **테스트 없이** 반영 |
| 코드 리뷰 보조 (놓친 엣지 케이스 지적) | 시크릿·API 키 취급, 카드정보 처리 경로 수정 |
| 레거시 정산 배치의 **흐름 문서화** | 규제 해석에 대한 최종 판단 |

특히 효과가 좋은 건 **"공격자 역할" 테스트 작성**입니다. "이 결제 API에 같은 멱등성 키로 동시에 두 번 요청하면? 타임아웃 후 재시도가 오면? 금액이 음수면?" — 사람이 놓치기 쉬운 시나리오를 에이전트에게 나열·구현시키면 방어망이 두꺼워집니다.

---

## 3. 안전장치 설계 — 하네스 차원의 통제

"조심해서 써야지"라는 다짐은 안전장치가 아닙니다. 통제는 **도구 설정 차원**에서 강제해야 합니다.

1. **권한 최소화**: 에이전트의 셸·파일 권한을 작업 디렉터리로 한정하고, 운영 환경 자격증명이 없는 환경에서만 실행합니다. 운영 DB 접속 정보가 든 셸에서 에이전트를 돌리는 것 자체가 사고입니다.
2. **금지 영역 명문화**: AGENTS.md에 "정산 금액 계산 모듈(`settlement-core`)은 제안만 하고 직접 수정 금지" 같은 규칙을 명시합니다.
3. **위험 명령 확인 게이트**: DB 마이그레이션, 배포, 외부 API 호출은 자동 승인하지 않고 사람 확인을 거치도록 도구의 권한 설정을 구성합니다 (human-in-the-loop).
4. **변경 추적**: AI가 만든 변경도 동일하게 PR·리뷰·CI를 통과시킵니다. "AI가 짰으니 빨리 머지"는 감사 관점에서 최악의 관행입니다.
5. **시크릿 격리**: 카드정보·API 키가 코드나 설정 파일로 에이전트 컨텍스트에 들어가지 않도록 경로를 차단합니다. 컨텍스트에 들어간 시크릿은 어디로 출력될지 통제할 수 없습니다.

이런 통제 구조를 설계하는 일 자체가 [하네스 엔지니어링]({% post_url 2026-05-07-harness-engineering %})의 영역입니다.

---

## 4. 실전 워크플로우 — 결제 취소 버그 수정 예시

1. **재현 테스트 먼저**: 버그를 재현하는 실패 테스트를 에이전트에게 작성시킵니다. 이 시점엔 프로덕션 코드를 건드리지 않습니다.
2. **원인 분석 보고**: "수정하지 말고, 원인과 영향 범위(이 코드 경로를 타는 다른 거래 유형 포함)를 먼저 보고하라."
3. **최소 수정**: 합의된 범위만 수정. diff가 합의 범위를 벗어나면 반려.
4. **불변식 검증**: 멱등성·대사 테스트 전체 통과 + 기존 회귀 테스트 통과.
5. **사람 리뷰 2인**: 고위험 모듈은 평소보다 리뷰 기준을 높입니다.

---

## 5. 마치며

결제 시스템에서 AI를 못 쓰는 게 아닙니다. **테스트·문서화·분석이라는 "안전한 표면적"이 오히려 넓은 도메인**입니다. 다만 핵심 로직과 운영 환경으로 가는 길목마다 사람의 게이트를 시스템으로 박아두는 것 — 그것이 속도와 안전을 동시에 얻는 방법입니다.
